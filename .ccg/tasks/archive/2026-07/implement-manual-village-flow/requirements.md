# 砍掉地图定位，改为村维度手动扫楼流程

## 需求摘要
当前定位/第三方地图 API 不稳定且有成本，MVP 改为完全离线、手动记录。核心对象从“地图上的房源点”调整为“村/片区扫楼项目”，用户可以提前创建城中村，现场进入该村，一栋一栋记录楼栋与房源。

## 已确认决策
- 采用三层结构：Village（村/片区） -> Building（楼栋/入口） -> HouseRecord（具体房源/房间）。
- 楼栋作为独立扫楼过程记录存在：即使没有房源，也允许保存楼栋，并记录“已扫无房/联系中/需复访/放弃/有房源/未扫”等状态。
- 快速新增房源采用“村必选，楼栋可后补/未分楼栋”的规则：房源必须归属某个村，但允许先暂存到该村的未分楼栋区，后续再归到具体楼栋。
- 去地图后，通勤/周边/环境信息采用“村级默认，房源级可覆盖”：村保存默认通勤分钟、交通备注、周边标签/评分、环境判断；房源只记录特殊覆盖项。
- 地图相关能力采用“产品上彻底砍掉，技术上先业务下线、后续分阶段清理底层遗留”的策略：第一阶段用户看不到地图/定位/经纬度/刷新/失败，第二阶段再逐步删 MapRepository/Amap/BFF map/map_snapshot 等遗留。
- 一级导航采用：首页 / 房源 / 对比 / 我的。首页是村列表/扫楼项目入口；房源 Tab 保留为全局房源复盘列表；对比 Tab 保留；我的 Tab 保留。
- 首页采用“继续扫楼卡片 + 村列表进度统计”：顶部展示最近/正在扫的村与继续按钮，下方展示全部村及楼栋数、房源数、候选数、待复访数、最近扫楼时间等统计。
- 村详情页以“楼栋扫楼工作台”为主，而不是以房源列表为主：村概览、快捷操作、楼栋状态统计、楼栋列表、未分楼栋房源、村内候选入口。
- 楼栋状态采用“固定枚举 + 标签/备注补充”：状态固定为未扫/已扫无房/有房源/联系中/需复访/已放弃；标签和备注用于表达电话没人接、晚上再来、门禁进不去、环境差、贴了招租、疑似中介、管理处问等特殊情况。
- 快速记录保持“月租 + 拍照 + 房型”为核心，同时增加“楼栋/入口”和“门牌/房号”两个可选快捷字段：村由入口带入且必选，楼栋可选/可新建/可后补，门牌本地加密。
- 村采用固定生命周期状态：准备中/扫楼中/暂停/已完成/已归档；首页排序优先扫楼中，其次准备中、暂停、已完成、已归档。
- 楼栋需要支持独立照片/附件，不再只能把照片挂在房源下；楼栋可挂入口、招租牌、楼道、门禁、环境等照片，房源继续挂房间、厨卫、窗外、水电表、合同等照片。实现建议采用通用 ownerType + ownerId 照片归属模型，支持 house/building/village。
- 删除策略采用“默认禁止误删，优先归档/放弃；强制删除需要危险确认”：有下级数据的村/楼栋默认不能直接删除，建议归档或标记放弃；强制删除必须二次确认并展示影响数量。
- 旧数据迁移采用自动创建“未分组”村：所有没有 villageId 的旧房源挂到“未分组”，buildingId 留空，显示为未分楼栋房源，用户后续可手动迁移到具体村/楼栋。

## 推荐方案
数据层级采用：Village（村/片区） -> Building（楼栋/入口） -> HouseRecord（具体房源/房间）。首页优先展示村列表，进入村后展示楼栋扫描进度和房源列表；新增房源必须归属某个村，可选归属某栋楼。

## 关键决策
- 砍掉 GPS、地图 SDK、地图 BFF、POI/通勤自动刷新。
- 通勤/周边改为手动字段：村级通勤分钟、交通备注、周边标签/主观评分；房源可覆盖。
- 保留成本、房屋质量、风险、照片、Checklist、对比导出等本地核心能力。
- 现有经纬度/地图快照字段实现时优先隐藏不用，避免迁移风险；后续确认无数据后再物理删除。
