# v1.1 正式版测试需求提示词（优化版）

> 日期：2026-07-07（Asia/Shanghai）  
> 事实源：`../租房扫楼产品PRDv1.1.md`  
> 适用对象：foundHouse Flutter 移动端 v1.1「村 / 楼栋 / 房源」手动扫楼流程

## 1. 原始需求

制定完整的 v1.1 正式版测试用例，真机测试用例需要覆盖全流程、每个功能、多维度、多入口。先优化需求提示词，然后根据提示词制定测试用例文档，再依据文档进行完整真机测试，汇总测试结果并生成测试报告。

## 2. 优化后的可执行提示词

请基于 `租房扫楼产品PRDv1.1.md`，为 foundHouse Flutter 移动端 v1.1 制定一套正式版 QA 验收方案，并在 Android 真机上执行可自动化的核心用例，最终输出测试报告。

### 2.1 测试目标

验证 v1.1 是否完成从「地图/定位驱动」到「村 / 楼栋 / 房源手动扫楼」的主流程切换，并确保离线记录、复盘、评分、对比、隐私脱敏等租房决策闭环可用。

### 2.2 必测范围

1. **一级导航与信息架构**：默认进入「首页」；一级 Tab 为「首页 / 房源 / 对比 / 我的」；首页不是地图页。
2. **村维度**：空状态、新增村、多村展示、继续扫楼、村统计、进入村详情。
3. **楼栋维度**：新增楼栋、楼栋无房源也能存在、六种状态（未扫 / 无空房 / 有空房 / 联系中 / 待复访 / 放弃）、楼栋入口记录房源。
4. **快速记录维度**：必须有 `villageId`；`buildingId` 可选；覆盖首页村卡片、村详情、楼栋卡片、缺失村负向入口；录入月租、楼栋备注、门牌/房号、房型、拍照/相册入口。
5. **房源列表与详情**：列表空态/非空态、村筛选、排序入口、进入详情、基础信息/费用/房屋/联系人/备注自动保存、Checklist 与风险摘要。
6. **评分与硬筛**：评分展示、红线淘汰、警告风险、评分详情路由/页面可用性（如入口存在）。
7. **对比与导出**：不足 2 套提示、选择 2 套后展示对比表、风险/淘汰行、导出入口、默认脱敏提示。
8. **设置与隐私**：「我的」页入口、偏好保存、隐私说明、本地优先、敏感字段加密、导出默认隐藏联系人/门牌/详细地址。
9. **旧地图能力回归**：主路径不得出现「扫楼地图」「定位失败」「经纬度」「高德」「第三方地图 API」等旧方案文案；主流程不得依赖地图/定位权限才能记录。
10. **真机稳定性**：Android 设备检测、冷启动、输入法、滚动、小屏可点击、返回栈、系统 UI 边界（相机/相册/分享面板）。

### 2.3 覆盖维度

- **流程维度**：扫楼前准备 → 现场逐栋扫楼 → 快速记录 → 回家复盘 → 评分/筛选 → 对比/导出 → 隐私确认。
- **入口维度**：首页、村详情、楼栋卡片、房源列表、设置页、直接路由/异常入口。
- **数据维度**：0 个村 / 1 个村 / 多个村；0 栋 / 多栋；0 房源 / 多房源；有楼栋归属 / 未分楼栋；有风险 / 无风险。
- **状态维度**：楼栋 6 状态；房源 active / shortlisted / rejected 或带 blocker；偏好未设置 / 已设置。
- **设备维度**：真机 Debug 测试、输入法收起、滚动命中、系统权限/系统页面可观察。

### 2.4 验收证据要求

- 每条自动化/命令型验证必须记录命令、设备、时间和结果。
- 真机自动化至少运行：`flutter test -d <device> integration_test/full_device_flow_test.dart` 与正式补充脚本。
- 静态与回归至少运行：`flutter analyze`、`flutter test`。
- 对无法由 Flutter integration test 稳定接管的系统 UI（相机、相册、分享面板），必须在测试用例与报告中标注为「手工系统 UI 项」，不得把未执行手工项写成已自动化通过。
- 任何失败、阻塞、未执行项都要给出原因、影响和建议。

### 2.5 通过标准

- P0/P1 自动化用例全部通过。
- 无阻断型缺陷：不能出现无 `villageId` 房源、首页仍是地图页、主流程要求定位、房源无法保存、对比/隐私关键链路不可用等问题。
- P2 手工系统 UI 项可以作为发布前补充清单，但报告必须明确未自动化边界。
- `flutter analyze` 与 `flutter test` 通过。
